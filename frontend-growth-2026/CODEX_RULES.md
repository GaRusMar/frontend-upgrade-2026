# Правила работы с Codex

## Главный принцип

Codex ускоряет работу, но не заменяет мышление.

Если я не могу объяснить код — я не коммичу его.

---

# Codex можно просить

- создать структуру проекта
- создать пустые файлы
- сверстать базовый HTML/CSS
- подготовить README-шаблон
- написать commit message
- объяснить ошибку
- найти баг
- предложить рефакторинг
- сделать code review
- сгенерировать тестовые данные
- подготовить чеклист проверки

---

# Codex нельзя просить

- полностью решить учебную задачу
- написать всю JS/TS/React-логику за меня
- писать бизнес-логику без моего участия
- исправлять код без объяснения причины
- коммитить код, который я не понимаю
- принимать diff без чтения

---

# Что я пишу сам

- состояние
- обработчики событий
- функции
- условия
- работу с массивами
- работу с объектами
- render-логику
- формы
- hooks
- работу с API
- бизнес-логику

---

# Codex workflow

1. Попросить Codex создать структуру файлов.
2. Самому написать основную логику.
3. Запустить проект в браузере.
4. Если ошибка — сначала самому прочитать консоль.
5. Потом попросить Codex объяснить ошибку, но не исправлять сразу.
6. Самому внести исправление.
7. Попросить Codex сделать code review.
8. Прочитать diff.
9. Закоммитить только понятный код.

---

# Перед коммитом я отвечаю

1. Что изменилось?
2. Почему так сделано?
3. Где хранится состояние?
4. Где бизнес-логика?
5. Как проверить работу?
6. Что может сломаться?
7. Какие есть альтернативы?
